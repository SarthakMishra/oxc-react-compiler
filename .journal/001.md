## Phase 78: Stable IdentifierId Refactor

**Date:** 2026-03-16
**Task:** .todo/stable-identifier-ids.md -- All phases (1–7)
**Upstream:** N/A (OXC-specific architectural fix)

The HIR was assigning a fresh `IdentifierId` to every `Place` created for a variable reference. A variable `x` referenced five times would have five distinct IDs, making it structurally impossible for downstream passes (freeze tracking, mutation, alias analysis) to correlate state across references to the same binding. This phase implements a stable-ID architecture across three coordinated sub-phases.

**Phase 1 (Types & Builder):** Added `ssa_version: u32` to the `Identifier` struct to carry the SSA rename index separately from the base identity. Added a `binding_ids: FxHashMap<String, (IdentifierId, DeclarationId)>` registry to `HIRBuilder`; `make_named_place()` now looks up or inserts into this registry so all references to the same binding share one `IdentifierId`. Temporaries via `make_temp()` still get fresh IDs. Updated all 15 `Identifier` construction sites across 9 files to include `ssa_version: 0`. The `InstructionValue` size assertion was bumped from 256 to 264 bytes to reflect the added field. **Phase 2 (SSA Rewrite):** Rewrote `enter_ssa.rs` to operate on `(IdentifierId, version)` pairs: version stacks are `FxHashMap<IdentifierId, Vec<u32>>`, `fresh_ssa_name()` became `fresh_ssa_version()` returning a `u32`, and phi operand matching uses the base `IdentifierId` directly. Removed three helper functions (`find_max_identifier_id`, `find_declaration_id_for_var`, `find_name_for_var`) that existed solely to work around the fresh-ID scheme. Rewrote `eliminate_redundant_phi.rs` to use `SsaId = (IdentifierId, u32)` tuples in its replacement map instead of `IdentifierId → IdentifierId`. **Phase 3–7 (Verification):** Audited all downstream passes (inference, reactive scopes, validation, optimization, codegen). No changes were needed: maps keyed by `IdentifierId` now have one entry per variable (correct semantics), temporaries retain unique IDs (no naming collision in codegen), and mutable ranges become slightly more conservative (wider) but semantically correct. All 160 unit tests pass; conformance holds at 1293 known failures (no regressions — this is a foundational correctness change, not a conformance fix).

## Phase 77: Hybrid Effects+Instruction Frozen-Mutation Validator Rewrite

**Date:** 2026-03-16
**Task:** .todo/false-bailouts.md -- Frozen-mutation false positives
**Upstream:** `src/Validation/ValidateNoMutationAfterFreeze.ts`

Rewrote `validate_no_mutation_after_freeze` from a 925-line name-based frozen tracking implementation to a ~335-line hybrid approach. The new design has two complementary layers: (1) effects-based detection using `MutateFrozen` effects emitted by the abstract interpreter during `infer_mutation_aliasing_effects`, and (2) targeted instruction-level checks for edge cases that the effects layer cannot express. The effects layer handles the common case of direct mutations on frozen values. The instruction-level layer covers three specific patterns: `MethodCall` receiver mutations (which cannot be expressed as effects without changing scope analysis for all method calls), `PropertyStore` on frozen targets, and nested function bodies that close over frozen values.

Hook return value pre-freezing and hook call capture freezing were also refined. The root cause of the previous over-flagging was the fresh-ID-per-reference architecture: because each reference to a variable gets a fresh abstract heap ID, the heap cannot link freeze state across references without a pre-linking step. The earlier attempt to pre-link by name forced all IDs to `Mutable` in the lattice, causing cascading false positives in other passes. The hybrid approach sidesteps this by using effects (which are already resolved to the correct IDs by the abstract interpreter) for the common case and falling back to instruction-level name matching only for the three patterns that genuinely require it. Additionally, the `Apply` fallback in `infer_mutation_aliasing_effects` emits `MutateTransitiveConditionally` for all arguments including frozen params; the validator now skips conditional mutations for `CallExpression`/`MethodCall` instructions to avoid those false positives. Net result: frozen-mutation false-positive bail-outs dropped 158 → 89 (−69), total bail-outs 312 → 258 (−54). Conformance moved 435 → 424 (−11): 4 real regressions requiring alias/phi/derivation-chain tracking to fix, and 7 pre-existing divergences that were previously masked by the false bail-outs and are now visible.

## Phase 75: `validate_no_unsupported_nodes` Pass + HIR Unsupported Pattern Detection

**Upstream:** `compiler/packages/babel-plugin-react-compiler/src/HIR/BuildHIR.ts`

Implemented the `validate_no_unsupported_nodes` validation pass, which walks all HIR instructions looking for `UnsupportedNode` values and emits a `CompilerError::todo()` for each one, causing the pipeline to bail out. This matches upstream `BuildHIR.ts` behavior where unsupported syntax patterns are rejected with a `TodoDiagnostic` at lowering time rather than silently compiled. The pass is wired at pipeline Phase 0 (pre-SSA, before any analysis runs) so that the error surfaces immediately without partial analysis artifacts.

Three new unsupported-pattern detection sites were added to `hir/build.rs`. All three patterns already had upstream `BuildHIR.ts` counterparts that throw `TodoException`.

In `validate_no_global_reassignment.rs`, the JSX prop safety heuristic was narrowed: previously all JSX prop values were treated as safe callback contexts, but the correct upstream behavior is to only suppress the reassignment warning for event-handler props matching the `onXxx` pattern. An unnecessary `FxHashSet::clone()` in the recursive `check_inner_scopes` call was eliminated by passing the existing set by reference. The `MetaProperty` format-string allocation in `build.rs` was also cleaned up — the string was being allocated and immediately dropped without the result being used.

Four previously-failing conformance fixtures began passing: `new-target.js`, `for-await.js`, and two getter/setter object property fixtures. Two pre-existing failures that were incorrectly omitted from `known-failures.txt` were added back. Net conformance: 430 → 434/1717 (+4 fixtures, 0 panics, 0 unexpected divergences).

## Phase 73: `rename_variables` Pass — Scope Output Variable Renaming

**Date:** 2026-03-14
**Task:** .todo/codegen-structure.md -- scope output variable renaming
**Upstream:** `compiler/packages/babel-plugin-react-compiler/src/ReactiveScopes/CodegenReactiveFunction.ts`

Implemented the `rename_variables` pass in `prune_scopes.rs`, which was previously a TODO stub. The pass assigns sequential temp names (`t0`, `t1`, …) to reactive scope declaration outputs, matching the upstream `CodegenReactiveFunction.ts` behavior. For each scope whose output declarations are eligible for renaming, the pass renames the binding to the next temp index and emits a `const originalName = tN` assignment immediately after the scope block so downstream code continues to reference the original name.

The implementation is organized into several helpers. `scan_max_temp_counter` / `scan_max_in_block` walk all existing instructions to find the highest `tN` suffix already in use, preventing collisions with temps generated in earlier passes. `temp_name_index` / `update_max_temp` parse and track those indices. `can_rename_scope_decl` is the main eligibility predicate: a declaration is skipped if it already has a temp name, is a reassignment target, or is read within the scope body (which would require the alias assignment to precede the read — upstream defers this to a later lowering step). `check_rename_eligibility` walks the scope body counting lvalue writes, reads, and detecting reassignment kinds using `InstructionKind`. `count_reads_in_value` and `count_reads_in_terminal` provide exhaustive `InstructionValue` and terminal-operand visitors that count name references, including `Switch` case tests. The pass required adding `InstructionKind` and `Param` to the import block. Two newly passing conformance fixtures were removed from `known-failures.txt`: `array-pattern-params.js` and `object-pattern-params.js`. Conformance moved from 397 → 399/1717 (+2, 0 regressions).

## Phase 71: Interleaved Fixpoint Loop for InferMutationAliasingEffects

**Date:** 2026-03-14
**Task:** .todo/over-memoization-bailout.md -- Gap 5: Effect Refinement
**Upstream:** `compiler/packages/babel-plugin-react-compiler/src/Inference/InferMutationAliasingEffects.ts` (worklist loop pattern)

Refactored `infer_mutation_aliasing_effects` from a sequential 4-phase architecture (collect raw effects → refine → apply to heap → store) into an interleaved fixpoint loop matching the upstream worklist pattern. Each iteration now performs all four steps per-instruction in sequence: compute raw effects, refine using the current heap state, apply to the heap, then store the refined effects on the instruction. The loop repeats over all instructions until no heap state changes are observed, at which point the result has converged.

To enable change tracking, all `AbstractHeap` mutating methods (`create`, `alias`, `capture`, `mutate`, `freeze`) were changed to return `bool` — `true` if the method modified heap state, `false` if the heap was already in the target state. A `merge_value_kinds()` lattice join function was added to support idempotent re-creation: when a place is created a second time with a different `ValueKind`, the function computes the join of the two kinds (`Frozen` ∨ anything = `Frozen`, `Primitive` ∨ `Primitive` = `Primitive`, otherwise `MutableObject`) and returns whether the existing entry was promoted. The dead `Apply` arm in `process_effect_for_heap` was removed — `Apply` effects are fully handled by `refine_effects()` before `process_effect_for_heap` is called, so the arm was unreachable. Documentation comments were added to both the convergence loop and `merge_value_kinds()` explaining the lattice semantics and termination proof (finite place set, monotone join, bounded height). Conformance held at 385/1717 with zero regressions — existing fixtures already converge in a single pass, so the fixpoint loop exits immediately after the first iteration for all current test cases.

## Phase 70: Function Signature Resolution for Apply Effects (Iteration 24)

**Date:** 2026-03-14
**Task:** .todo/over-memoization-bailout.md -- Gap 5: Effect Refinement
**Upstream:** `compiler/packages/babel-plugin-react-compiler/src/Inference/InferMutationAliasingEffects.ts` (`applyEffect()`), `compiler/packages/babel-plugin-react-compiler/src/Inference/AnalyseFunctions.ts`

`analyse_functions.rs` was completely rewritten to produce a `FxHashMap<IdentifierId, FunctionSignature>` (one entry per function place) rather than the previous flat effect list. The new implementation walks each function's HIR blocks and extracts per-parameter effects: for each parameter it collects all effects that reference that parameter's place, resolving aliases through `StoreLocal`/`LoadLocal`/`TypeCast` chains before attribution. A fixpoint loop propagates signatures transitively — when an inner function calls an outer one whose signature is already known, the callee's parameter effects are inlined into the caller's signature — continuing until no new entries are added. This matches the `AnalyseFunctions.ts` upstream algorithm that builds a stable fixpoint map before any `applyEffect` resolution begins.

The `refine_effects()` phase (Phase 3.5) in `infer_mutation_aliasing_effects.rs` implements the upstream `applyEffect()` logic. It runs after initial effect collection and before the final range update. `Apply` effects are the primary target: when the called function has a known signature entry in `fn_signatures`, each argument is mapped per-parameter — `Mutate` argument effect becomes `MutateTransitively`, `Capture` stays `Capture`, `Freeze` stays `Freeze`, and alias-to-return effects produce a new `Alias` between the argument place and the call's return place. When no signature is available the conservative path emits `MutateTransitiveConditionally` for every argument, matching upstream's fallback. A `value_kind()` method was added to `AbstractHeap` to power the remaining refinements: `CreateFrom`/`Assign` from a Primitive or Global collapse to `Create(Primitive)`; from a Frozen source they become `Create(Frozen)` + `ImmutableCapture`. `Capture` of a Primitive or Global is dropped (primitives cannot be aliased); of a Frozen place it becomes `ImmutableCapture`. `MutateConditionally`/`Mutate` of a Frozen place emit `MutateFrozen`. The `aliasing_effects.rs` instruction handler was updated to accept `fn_signatures` and perform signature lookups at `CallExpression`, `NewExpression`, and `TaggedTemplateExpression` sites, passing the resolved `FunctionSignature` (or `None`) into the `Apply` effect that `refine_effects()` later processes. `pipeline.rs` wires the new data flow: `analyse_functions` runs first and its output map is threaded into `infer_mutation_aliasing_effects`. Conformance moved from 370 → 385/1717 (+15 fixtures, 22.4%), 0 panics.

## Phase 69: `refine_effects()` — Apply Resolution and Value-Kind-Based Effect Refinement

**Date:** 2026-03-14
**Task:** .todo/over-memoization-bailout.md -- Gap 5
**Upstream:** `compiler/packages/babel-plugin-react-compiler/src/Inference/InferMutationAliasingEffects.ts` (`applyEffect()`)

Added the `refine_effects()` phase (Phase 3.5 in `infer_mutation_aliasing_effects.rs`) which implements the upstream `applyEffect()` logic. This pass runs after the initial effect collection and before the final range update. It iterates over all collected effects and resolves or refines them based on the abstract heap's current value-kind knowledge, replacing high-level abstract effects with concrete lower-level ones.

The most structurally significant case is `Apply` resolution. When the called function has a known signature, each argument effect is mapped per-parameter: `Mutate` → `MutateTransitively`, `Capture` → standard `Capture`, `Freeze` → `Freeze`, and alias-to-return effects produce a new `Alias` entry between the argument and return place. When no signature is available, the conservative path emits `MutateTransitiveConditionally` for every argument — matching upstream's fallback. ## Phase 68: Graph-Based BFS Mutation Propagation in InferMutationAliasingRanges

**Date:** 2026-03-14
**Task:** .todo/scope-merging.md -- Mutation aliasing ranges rewrite
**Upstream:** `compiler/packages/babel-plugin-react-compiler/src/Inference/InferMutationAliasingRanges.ts`

Rewrote `infer_mutation_aliasing_ranges.rs` from a flat one-level alias traversal (~315 lines) to a graph-based BFS mutation propagation algorithm (~715 lines). The previous implementation used three parallel maps — `alias_map`, `mutation_map`, and `last_use_map` — with a single-step alias resolution that could not propagate mutations transitively through multi-hop alias chains (e.g., A aliases B captures C, mutation of C never reached A). The new implementation builds a directed `AliasingGraph` during the instruction scan phase and then processes all collected mutations via BFS in a deferred second phase.

The central data structure is `AliasingGraph` with `Node` entries that store forward edges plus typed backward sets (`aliases`, `created_from`, `captures`, `maybe_aliases`). Mutations are collected as `PendingMutation` work items and processed by the `mutate()` BFS function, which follows edges with temporal index guards — only edges created before the mutation's instruction index are followed, matching the upstream rule that aliasing created after a mutation cannot retroactively extend the mutable range. Phi-node back-edges are handled via a `pending_phis` map: when BFS reaches a phi node that was added before its predecessor blocks were processed, the phi edge is deferred and replayed after all blocks complete. A `phi_ids` set prevents backward alias traversal through phi nodes encountered via forward edges, preventing infinite cycles. For backward compatibility, `creation_map` and `last_use_map` from the original implementation were retained, with `creation_map` extended to cover `CreateFunction` and `CreateFrom` instructions missed in the previous version. All `unwrap()` calls were eliminated via a unified `add_edge()` helper method that returns `Option` and propagates through `?`. A P0 edge-ordering bug (BFS loop used `break` instead of `continue` for deferred phi edges) was fixed during code review, and a P2 suspicious `range.start` mutation was removed. Conformance held at 370/1717 with zero regressions.

## Phase 50: MergeOverlappingReactiveScopesHIR — 3-Phase Active-Scope-Stack Rewrite

**Date:** 2026-03-14
**Task:** .todo/memoization-structure.md -- Sub-task 4a: active-scope-stack overlap detection (Pass 42)
**Upstream:** `compiler/packages/babel-plugin-react-compiler/src/ReactiveScopes/MergeOverlappingReactiveScopesHIR.ts`

Rewrote `merge_overlapping_reactive_scopes_hir()` in `merge_scopes.rs` from a simple flat-range sort-and-merge into a 3-phase algorithm that matches upstream's semantics. The old approach sorted all scope ranges by start position and merged any two ranges that geometrically overlapped, which was both over-eager (it merged scopes that shared an edge but had no data-flow relationship) and under-specified (it did not account for cross-scope mutation). The new algorithm mirrors `MergeOverlappingReactiveScopesHIR.ts` exactly.

Phase 1 builds four index structures from a single pass over all instructions: `scope_starts` (InstructionId → ScopeId), `scope_ends` (InstructionId → ScopeId), `place_scope` (PlaceId → ScopeId for mutated places), and `mutated_at` (InstructionId → Vec<PlaceId> for instructions that mutate a place). A `BTreeSet` over instruction IDs ensures the walk is in strict program order regardless of HIR storage order. Phase 2 maintains an active-scope stack (`Vec<ScopeId>`) and a `DisjointSet<ScopeId>` (reusing the existing `utils/disjoint_set.rs`). At each scope-end event, if the ending scope is not the stack top, upstream's invariant says it must overlap with the top scope, so they are unioned. At each instruction, every mutated place is checked: if the place belongs to a scope different from the current top scope, those two scopes are also unioned. Phase 3 collects canonical roots from the DSU, computes merged start/end ranges over all member scopes, then rewrites every `Identifier` annotation in the HIR's scope map and block scope assignments to use the canonical ScopeId. All collections use `FxHashMap`/`FxHashSet`; `BTreeSet` is used only for deterministic instruction-ID iteration. Every `unwrap()` was replaced with `Option` pattern matching for defensive correctness.

Ten snapshot files were regenerated across four test suites (component-with-conditional, component-with-derived-value, hook-with-use-state, semantic-derived-values, fixture per-pass snapshots for basic-memoization, conditional-rendering, hook-dependencies, and multiple-scopes). ## Phase 47: validate_no_mutation_after_freeze — Hook-Return Pre-Freeze, Capture Tracking, Nested Function Scanning

**Date:** 2026-03-13

The `react/jsx-runtime` import was removed from both `generate_import_statement` and `apply_compilation`, eliminating the now-spurious runtime import. 23 snapshot files were updated to reflect JSX syntax output. Conformance stays at 304/1717 because the conformance comparison normalizes both sides through the OXC JSX transform before diffing, so JSX preservation is structurally invisible to the counter — but it is required for correctness alignment with upstream.

## Phase 35: Recursive Temp Use-Counting for Cross-Scope Expression Inlining

**Date:** 2026-03-13
**Task:** .todo/memoization-structure.md -- Gap 1: Temp variable inlining pass
**Upstream:** `compiler/packages/babel-plugin-react-compiler/src/ReactiveScopes/CodegenReactiveFunction.ts`

`build_inline_map` previously counted temp uses only at the flat block level, iterating over instructions in the current block's instruction list. Uses inside nested `ReactiveScopeBlock` bodies and `ReactiveTerminal` child blocks (if/switch/while/do-while/for/for-of/for-in/try/label) were invisible to it, so any temp whose only use was inside a scope body appeared to have `use_count=0` and was either dropped or not inlined — producing unnecessary `let t0;` declarations instead of inlined expressions. This was a silent correctness gap: the output was semantically wrong for temps that existed solely to feed a cached scope body.

The fix adds two recursive helpers: `count_temp_uses_recursive` walks the entire instruction subtree of a `ReactiveStatement` slice (recursing into `ReactiveScopeBlock` bodies and all terminal child blocks via `visit_terminal_child_blocks`), and `visit_terminal_child_blocks` covers all nine terminal variants that carry child statement lists. A two-tier counter — `total_count` (all uses, any depth) and `flat_count` (uses only in the current flat block) — drives a safety-aware inlining gate: the single-use path uses `total_count` to correctly detect inlineable temps that live only inside scopes, while the call-safety path uses `flat_count` to prevent side-effecting call expressions from being inlined into cache-guarded scope bodies that may be conditionally skipped. All `HashMap`/`HashSet` uses in `codegen.rs` were also migrated to `FxHashMap`/`FxHashSet` for consistency with the rest of the codebase.

Note: Babel's codegen (`CodegenReactiveFunction.ts`) never emits SSA temporaries — it inlines expressions during tree printing. Our port flattens to SSA in BuildHIR and must re-collapse in codegen; this recursive counting pass is the correctness foundation for that re-collapse. Conformance stays at 304/1717 because temp inlining alone does not resolve the structural differences driving the remaining failures (Gap 2 JSX preservation, Gap 5 sentinel scopes, Gap 6 over-scoped deps), but it is a required prerequisite for all three.

## Phase 14: Codegen Correctness & Memoization Pipeline Foundation

**Date:** 2026-03-11
**Task:** `.todo/backlog.md -- Codegen correctness (Priority 1) and Memoization pipeline (Priority 2)`
**Upstream:** `compiler/packages/babel-plugin-react-compiler/src/ReactiveScopes/PruneReactiveScopes.ts`, `InferReactiveScopeVariables.ts`, `PropagatePhiDependencies.ts`, `BuildReactiveFunction.ts`, `compiler/packages/babel-plugin-react-compiler/src/Inference/InferReactivePlaces.ts`, `InferMutableRanges.ts`

Two inter-related bug clusters were fixed in this iteration. The first was a codegen naming gap: `promote_used_temporaries` only walked lvalue identifiers, leaving operand references (the RHS of instructions, terminal operands, and scope dependency/declaration identifiers) with their name field still set to `None`. At emission time, `place_name()` hit the `_t{id}` fallback for those unnamed operands, producing broken variable names with a leading underscore. The fix added `promote_places_in_value()`, `promote_places_in_terminal()`, `promote_place()`, and `promote_identifier()` helpers in `prune_scopes.rs` that walk the full instruction operand graph. All `format!("_t{}", id)` fallback strings in `codegen.rs` were also changed to `format!("t{}", id)` as a belt-and-suspenders measure so that any remaining unnamed temporaries emit valid JS identifiers.

The second cluster addressed the memoization pipeline producing no usable scopes. Four sub-fixes were required: (1) `infer_reactive_scope_variables.rs` was discarding the return value of scope creation and never writing scopes back to identifier `.scope` fields — added a Phase 4 assignment pass; (2) `infer_reactive_places.rs` was not seeding function parameters as reactive — added `DeclareLocal` entry-block scanning — and `has_reactive_operand()` was only handling a small subset of instruction variants, causing reactive propagation to stall; (3) `infer_mutation_aliasing_ranges.rs` built mutable ranges from mutation sites only, but reactive scope inference requires ranges that span to the last use — added a `last_use_map` pass that extends each range's end to cover the final operand reference; (4) `propagate_dependencies.rs` populated `ReactiveScope.dependencies` but never populated `ReactiveScope.declarations` — added a Phase 3 reverse-use pass using an `operand_consumers` map. Additionally `build_reactive_function.rs` gained `build_scope_block_only()` to suppress instruction duplication when processing `Terminal::Scope` by not following Goto terminals into the fallthrough block. An E2E snapshot test `test_e2e_memoization` was added for a `Counter` component with `useState`; it documents the current partial state where cache allocation is emitted but scope boundary alignment still captures discriminant markers rather than computation instructions.

## Phase 3: HIR Foundation Types and Utilities

**Date:** 2026-03-11
**Task:** `.todo/hir-types.md`, `.todo/environment.md`, `.todo/utils.md`
**Upstream:** `src/HIR/HIR.ts`, `src/HIR/Environment.ts`, `src/HIR/ObjectShape.ts`, `src/HIR/Globals.ts`

Implemented all core data structures for the compiler's internal representation in `hir/types.rs` (~1000 lines). ID newtypes (`BlockId`, `IdentifierId`, `ScopeId`, `InstructionId`) use newtype wrappers over `u32` with `Display`/`From` impls. `Place` pairs an `IdentifierId` with a `ProjectionPath` for member-access lowering. `InstructionValue` is a 40-variant enum covering all lowered expression forms: `LoadLocal`, `StoreLocal`, `LoadGlobal`, `BinaryExpression`, `UnaryExpression`, `CallExpression`, `MethodCall`, `NewExpression`, `PropertyLoad`, `PropertyStore`, `ComputedLoad`, `ComputedStore`, `JsxElement`, `JsxFragment`, `JsxText`, `ArrayExpression`, `ObjectExpression`, `Primitive`, `TemplateLiteral`, `TaggedTemplateExpression`, `LogicalExpression`, `ConditionalExpression`, `Phi`, `StartMemoize`, `FinishMemoize`, and others. `Terminal` is a 20-variant enum encoding all CFG control flow: `Goto`, `Branch`, `Switch`, `Return`, `Throw`, `Try`, `MaybeThrow`, `Sequence`, and labeled variants. `BasicBlock`, `HIR`, `HIRFunction`, and `ReactiveFunction` IR types round out the core structural types. The `ReactiveScope` type contains the scope's dependency set, output set, and mutable range.

`environment.rs` defines `EnvironmentConfig` with ~20 boolean flags governing which passes run and which validations are enabled, plus `CustomHookConfig` for user-defined hook shapes. `object_shape.rs` implements `ShapeRegistry` (a global registry of named shapes), `FunctionSignature` (parameterized call kind and return type), `PropertyShape` (property access shapes), and the `CallKind` enum distinguishing hook calls from regular calls. `globals.rs` implements `register_globals()` which populates the `ShapeRegistry` with shapes for `Array`, `Object`, `Math`, `JSON`, `console`, `String`, `Number`, `Promise`, React hooks (`useState`, `useReducer`, `useEffect`, `useRef`, `useMemo`, `useCallback`, `useContext`, `useId`, `useTransition`, `useDeferredValue`, `useSyncExternalStore`), and DOM globals. `disjoint_set.rs` implements union-find with path compression and union-by-rank. `ordered_map.rs` implements an insertion-order-preserving map using a `Vec` of key-value pairs with `O(n)` lookup — appropriate for the small maps used in scope dependency tracking.
